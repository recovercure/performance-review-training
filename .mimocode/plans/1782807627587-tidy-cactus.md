# 方案：从裸 LLM 调用升级到 LangChain + LangGraph

## 一、目标

- 用 **LangGraph StateGraph** 建模 4 步训练流程（显式状态机，支持条件分支与可恢复性）
- 用 **LangChain ChatPromptTemplate** 替换手动字符串拼接，支持模板复用与热切换
- 用 **LangChain LCEL** 链式编排 RAG 检索链路
- 保留现有 API 路由作为 fallback，新旧共存，逐步切换

## 三、LangGraph 状态图设计

### 3.1 状态 Schema

```ts
interface TrainingState {
  sessionId: string;
  employeeType: EmployeeType;
  employeeName: string;
  difficulty: Difficulty;
  ragEnabled: boolean;
  
  // 当前步骤 (1-4)
  currentStep: number;
  
  // 本轮对话历史（管理者 + AI员工）
  messages: BaseMessage[];
  // 仅当前步骤的消息
  stepMessages: BaseMessage[];
  
  // 各步骤分析结果
  analyses: Record<number, AnalysisResult>;
  
  // 知识库检索结果（可选）
  knowledgeContext: string;
  
  // 最终总结
  summary: Record<string, unknown> | null;
  overallScore: number;
  
  // 流程控制
  status: "in_progress" | "completed";
  nextAction: "chat" | "analyze" | "next_step" | "finish" | "retry_step";
}
```

### 3.2 图结构

```
                    ┌──────────┐
                    │  START   │
                    └────┬─────┘
                         │
                    ┌────▼──────┐
                    │ build_prompt│  ← 构造当前步骤的 System Prompt
                    └────┬──────┘
                         │
              ┌──────────▼──────────┐
              │   rag_retrieve?      │  ← 条件边：ragEnabled=true 时注入知识库
              └──────┬──────┬───────┘
                     │no    │yes
                     │      ▼
                     │  ┌──────────┐
                     │  │ search_kb │
                     │  └────┬─────┘
                     │       │
                     └───────┼───────┘
                             │
                     ┌───────▼───────┐
                     │  employee_chat │  ← 流式输出 SSE（保持现有接口协议）
                     └───────┬───────┘
                             │
                     ┌───────▼───────┐
                     │ coach_analyze │  ← 调用分析 LLM
                     └───────┬───────┘
                             │
                     ┌───────▼───────┐
                     │   decision     │  ← 根据评分与步骤决定下一步
                     └───┬───┬───┬───┘
                         │   │   │
          ┌──────────────┘   │   └──────────────┐
          ▼                  ▼                  ▼
   currentStep < 4     score < 50 且       currentStep === 4
   → loop next_step    step < 4            → finish → write_summary
                       → retry (重新练习本步)
```

### 3.3 节点定义

| 节点 | 职责 | 输入 | 输出 |
|------|------|------|------|
| `build_prompt` | 用 `ChatPromptTemplate` 构造 System Prompt | `state.employeeType, state.employeeName, state.step` | `state.messages`（插入 system msg） |
| `search_kb` | 调用 Supabase `search_knowledge()` + `formatKnowledgeContext` | `state.ragEnabled, messages.last` | `state.knowledgeContext` 注入 prompt |
| `employee_chat` | 流式调用 LLM 模拟员工回复 | `state.messages ` | SSE stream（返回前端） |
| `coach_analyze` | 非流式调用 LLM 做评分 | `state.stepMessages` | `state.analyses[currentStep]` |
| `decision` | 纯逻辑节点，无 LLM 调用 | `state.analyses, currentStep` | `state.nextAction` |
| `write_summary` | 调用 LLM 生成总结报告 | `state.analyses` | `state.summary, state.overallScore` |

### 3.4 边 / 条件路由

```ts
function routeAfterEmployee(state: TrainingState): string {
  // 对话节点之后：如果需要分析 → coach_analyze，否则结束本轮
  return state.nextAction === "analyze" ? "coach_analyze" : "__end__";
}

function routeAfterDecision(state: TrainingState): string {
  if (state.nextAction === "finish") return "write_summary";
  if (state.nextAction === "retry_step") return "build_prompt";
  if (state.nextAction === "next_step" && (state.currentStep + 1) <= 4) return "build_prompt";
  return "__end__";
}
```

## 五、关键技术决策

### 5.1 Prompt 模板化

当前 `src/lib/prompts.ts` 三个函数 `buildEmployeeSystemPrompt / buildAnalysisSystemPrompt / buildSummaryPrompt` 用模板字符串拼接，搬到 `ChatPromptTemplate` 后：

```ts
// agents/prompts/employee.ts
import { ChatPromptTemplate } from "@langchain/core/prompts";

export const EMPLOYEE_PROMPT = ChatPromptTemplate.fromMessages([
  ["system", `你是一位专业的角色扮演AI...（现有内容）
## 你的角色设定
- 姓名：{employeeName}
- 员工类型：{employeeTitle}
- 性格特征：{personalityTraits}
...
## 当前面谈阶段
第{step}步 - {stepTitle}
...
{knowledgeContext}`],
  ["placeholder", "{chat_history}"],
]);
```

**收益**：变量注入由框架保证类型安全，`{knowledgeContext}` 可选，`chat_history` 用 `MessagesPlaceholder` 自动拼接。

### 5.2 RAG 链

当前 `chat/stream/route.ts` 手动调用 `searchKnowledge()` + `formatKnowledgeContext()`，改为 LCEL 链：

```ts
// agents/rag-chain.ts
import { RunnableSequence } from "@langchain/core/runnables";

const ragChain = RunnableSequence.from([
  {
    query: (input: { lastUserMessage: string }) => input.lastUserMessage,
  },
  // step: search
  async (input) => searchKnowledge(input.query, 3),
  // step: format
  (results) => formatKnowledgeContext(results),
]);
```

**收益**：可独立测试，可加入 fallback（检索空则跳过），未来可加 rerank 节点。

### 5.3 流式输出适配

当前 `llmStream()` 直接操作 `ReadableStream` + SSE，LangGraph 的 `streamEvents` 或 `stream` API 可以原生返回流。采用方案：

- **方案 A（推荐）**：LangGraph `stream()` 模式，每个节点产出的事件通过 SSE 透传到前端，前端不做改动
- **方案 B**：仅在 `employee_chat` 节点内部用 `ChatOpenAI.stream()` 逐 token 产出

选 **方案 A + B 结合**：LangGraph 整体用 `.stream()` 控制流程可见性，`employee_chat` 节点内部逐 token 流式返回员工消息。

### 5.4 新旧共存策略

新增 `/api/v2/chat/stream` 和 `/api/v2/analysis` 两条路由走 LangGraph：

- `/api/v2/chat/stream` → 调用 `graph.stream(state, { streamMode: "messages" })` 
- `/api/v2/analysis` → 调用 `graph.invoke()` 进行分析节点
- 旧路由 `/api/chat/stream`、`/api/analysis` 不删，保持现有行为
- 前端 `training/session/page.tsx` 加一个环境变量开关 `NEXT_PUBLIC_USE_LANGGRAPH`，默认关闭

### 5.5 可用开关控制

`.env` 新增：
```
# LangGraph 升级开关
USE_LANGGRAPH=false          # 服务端：控制 API 版本
NEXT_PUBLIC_USE_LANGGRAPH=false  # 客户端：前端是否走 v2 路由
```

### 5.6 ChatOpenAI 工厂

```ts
// lib/langchain-client.ts
import { ChatOpenAI } from "@langchain/openai";

export function createChatModel(options?: { temperature?: number }) {
  return new ChatOpenAI({
    apiKey: process.env.LLM_API_KEY,
    configuration: { baseURL: process.env.LLM_BASE_URL },
    model: process.env.LLM_CHAT_MODEL || "gpt-4o-mini",
    temperature: options?.temperature ?? 0.8,
  });
}

export function createAnalysisModel() {
  return new ChatOpenAI({
    apiKey: process.env.LLM_API_KEY,
    configuration: { baseURL: process.env.LLM_BASE_URL },
    model: process.env.LLM_ANALYSIS_MODEL || "gpt-4o-mini",
    temperature: 0.3,
  });
}
```

**关键点**：`ChatOpenAI` 的 `configuration.baseURL` 对应 OpenAI Node SDK 的 `baseURL` 选项，与当前 `fetch(${baseUrl}/chat/completions)` 协议完全相同。DeepSeek / OpenAI / 任意 OpenAI-compatible 服务都可用。

## 九、风险与注意事项

1. **LangGraph 的 `messages` 状态键是保留名**：建议自定义键名 `chatMessages` 避免冲突，或显式声明 `MessagesState`。
2. **流式模式选择**：`graph.stream()` 默认模式返回每个节点产出的完整 state 快照；如需逐 token 流，在 `employee_chat` 内部用 `model.stream()` 并通过自定义 streamMode 传出。
3. **环境变量读取**：LangChain 的 `ChatOpenAI` 默认读 `OPENAI_API_KEY`，我们显式传入 `apiKey` 避免干扰。
4. **tsup 打包兼容**：`@langchain/core` 和 `@langchain/langgraph` 都是 ESM/CJS 双包，tsup 的 `target: "node"` 下正常工作。
5. **不删旧代码**：旧 `prompts.ts`、`llm-client.ts`、旧 API 路由全部保留，只在 .env 开关控制下启用新的。
